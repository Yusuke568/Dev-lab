# 完全ヘキサゴナルアーキテクチャ移行計画書

## 1. 目的

本プロジェクトを、現在共存している「過渡期のレイヤードアーキテクチャ」から「完全なヘキサゴナルアーキテクチャ（ポート＆アダプタ）」へと統一・移行します。これにより、ビジネスロジックの独立性を最大化し、長期的な保守性とテスト容易性を確立します。

## 2. 現在の課題

- **パッケージの不統一**: `com.example`（旧）と `com.example.kintai`（新）で構造が異なり、開発者が迷う原因となっている。
- **エンティティの貧血化**: `com.example.entity` 配下のクラスが単なるデータコンテナ（Getter/Setterのみ）になっており、ドメイン知識がServiceに漏れ出している。
- **ポートの配置**: 外向きポート（Repository）が `application.port.out` だったり `domain.port.out` だったりと統一されていない。
- **コントローラーの所在**: `com.example.controller` 配下にActionが残っており、内向きアダプタとしての位置づけが曖昧。

## 3. 目標とする構造 (Target Architecture)

以下の構成をプロジェクト全体の標準とします。

```
com.example.<context>/
├── domain/
│   ├── model/         # 豊かなドメインモデル (Entity, Value Object)
│   └── port/
│       └── out/       # 外向きポート (Repository IF)
├── application/
│   ├── port/
│   │   └── in/        # 内向きポート (UseCase IF, Command)
│   └── service/       # ユースケース実装 (Transaction境界)
└── adapter/
    ├── in/
    │   └── web/       # 内向きアダプタ (Action, DTO, FrontController)
    └── out/
        └── persistence/ # 外向きアダプタ (PersistenceAdapter, SQL)
```

## 4. 移行フェーズ

### フェーズ1: 基盤と共通部分の整理

- **共通アダプタの移動**: `com.example.adapter` 配下の共通コンポーネント（TransactionManager等）を整理。
- **FrontControllerの移動**: `adapter.in.web` への配置。

### フェーズ2: 社員管理（Shain）ドメインの移行

- `com.example.entity.Shain` -> `com.example.shain.domain.model.Shain` への変換とロジック集約。
- ポート、サービス、アダプタを `com.example.shain` 配下へ再配置。

### フェーズ3: 有給管理（PaidLeave）ドメインの移行

- 社員管理と同様に `com.example.leave` (仮) 配下へ移行。

### フェーズ4: 認証・共通機能の移行

- `Login` 関連の機能を `com.example.auth` (仮) 配下へ移行。

### フェーズ5: 全体統合とクリーンアップ

- `com.example` 直下の古いパッケージ（controller, application, entity, dao, adapter）の削除。
- `DependencyFactory` の最終調整。

## 5. 移行の基本ルール

1. **ドメインを純粋に保つ**: `domain` パッケージは Java 標準ライブラリ以外の依存（JDBC, Servlet等）を一切持たない。
2. **DTOの徹底**: アダプタとアプリケーション層の間でドメインモデルを直接晒さず、DTOを介してやり取りする（特にWeb側）。
3. **テストの並行実装**: 移行する各ドメインに対し、ドメインモデルの単体テストとユースケースのモックテストを必ず実装する。

## 6. リスクと対策

- **デグレードのリスク**: 各フェーズ完了後に既存の結合テスト（または手動テスト）を実施し、機能を保証する。
- **作業量の増大**: ドメインごとにフェーズを分けることで、段階的なリリースとリスク分散を図る。
